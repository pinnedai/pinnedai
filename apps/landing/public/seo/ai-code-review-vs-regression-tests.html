AI code review vs. regression tests — and why you want both

CodeRabbit, Greptile, Qodo, and Copilot Workspace are great at reviewing your PRs. They post comments, flag bugs, suggest improvements. But review comments fade after merge. Pinned is different — it turns PR claims into permanent regression tests. Use them together: the reviewer comments, Pinned protects.

The two distinct workflows

AI PR reviewers and claim-based regression testing solve different problems at different points in the lifecycle:

Concrete scenario

A Cursor-generated PR claims: "Auth required on /api/admin/export."

The AI reviewer's response: "Looks good. Verified the middleware adds the auth check." Comment is posted. PR merged. Comment disappears into history.

Pinned's response: Generates tests/pinned/pr-1247-auth-required-api-admin-export.test.ts that fires an unauthenticated request to /api/admin/export and asserts 401/403. Auto-commits to the PR branch. PR merged with the test included. The test runs on every future commit forever.

Six months later, a refactor moves the route and the middleware no longer matches. The AI reviewer is silent — it doesn't know about PR #1247's claim. Pinned's test fails with: "This commit breaks a claim pinned in PR #1247: 'Auth required on /api/admin/export.'"

Why the artifact matters

The structural difference: AI reviewers produce advice. Pinned produces code that lives in your repo. Code persists; advice fades.

This isn't an accident — it's the architectural choice. Pinned's value-prop only works if the tests are durable artifacts. CodeRabbit's value-prop only works if their LLM is reviewing your specific PR right now. Different products. They don't compete.

Stacking them: the recommended workflow

  1. Use your favorite AI code reviewer (CodeRabbit, Greptile, etc.) for inline feedback on every PR.
  2. Use Pinned alongside it to convert any "this PR adds X" claim into a permanent test.
  3. If the AI reviewer catches a bug, fix it. If Pinned's CI later fails, you know the claim regressed.

The two workflows are additive: reviewer protects against bugs visible now; Pinned protects against regressions visible later.
