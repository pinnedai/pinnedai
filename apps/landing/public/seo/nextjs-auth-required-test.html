Next.js auth-required test from a PR claim

When a Next.js PR adds a new API route or modifies middleware, the most common silent regression is losing the auth check. Pinned catches it: write the claim in your PR description and a permanent CI test enforces it.

The Next.js auth-regression pattern

You add a new route at app/api/admin/export/route.ts. Your middleware matches /api/admin/:path* and adds the auth check. Tests pass. Production is fine.

Three months later, someone refactors the middleware matcher from /api/admin/:path* to /admin/:path* (dropping the /api prefix by mistake during a route reorganization). The admin export endpoint is now public. No existing test catches this because no test specifically asserts that /api/admin/export requires auth.

What Pinned generates

Add this to your PR description:

Auth required on /api/admin/export.

Pinned's GitHub Action commits this test to your repo:

// tests/pinned/pr-42-auth-required-api-admin-export.test.ts
import { describe, it, expect, beforeAll } from "vitest";

const PREVIEW_URL = process.env.PREVIEW_URL;
const ROUTE = "/api/admin/export";

describe("pinned: auth-required on " + ROUTE, () => {
  beforeAll(() => {
    if (!PREVIEW_URL) throw new Error("PREVIEW_URL required");
  });

  it("returns 401 or 403 when called without an Authorization header", async () => {
    const url = PREVIEW_URL + ROUTE;
    const res = await fetch(url, { method: "GET" });
    expect([401, 403].includes(res.status)).toBe(true);
  });
});

Works with both App Router and Pages Router

Pinned's pinned baseline command scans your repo and auto-detects new API routes in both Next.js routing modes:

  • App Router: app/api/admin/export/route.ts → suggests Auth required on /api/admin/export.
  • Pages Router: pages/api/admin/export.ts → suggests Auth required on /api/admin/export.
  • Src prefix: src/app/api/... also detected

So even if you forgot to add the claim to a past PR's description, npx pinnedai baseline on your existing repo will suggest pins for every public admin route you haven't yet protected.

The middleware refactor scenario, replayed

With Pinned installed, when the middleware refactor accidentally widens the matcher and lets /api/admin/export through unauthenticated, CI fails with:

FAIL  tests/pinned/pr-42-auth-required-api-admin-export.test.ts
  AssertionError: expected 401 or 403 with no auth header, got 200

═══ PINNED FAILURE — paste this into Claude Code / Cursor ═══

Fix the failing pinned claim:
  Claim: "Auth required on /api/admin/export"
  Original PR: pr-42
  Route: /api/admin/export
  Expected: unauthenticated GET to /api/admin/export returns 401 or 403
  Actual: returned 200

Restore the auth check on /api/admin/export. Likely candidates:
  - middleware.ts or src/middleware.ts (Next.js / Hono / generic)
  - The route handler file for /api/admin/export
  - Auth provider config (Clerk / Auth.js / Supabase / Lucia)
Preserve authenticated behavior. Do not modify this pinned test file.

Works with any auth provider

Pinned doesn't care what auth library you use — Clerk, Auth.js (NextAuth), Supabase Auth, Lucia, custom JWT, etc. The test asserts the external behavior (401/403 for unauthenticated requests), not the internal implementation. Swap your auth provider and the test keeps working.

Install

npx pinnedai init

Then add Auth required on /api/your-route. to your PR descriptions. Or run npx pinnedai baseline to see candidate pins from your existing routes.
