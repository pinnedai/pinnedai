Stripe webhook idempotency test — auto-generated from your PR

A Stripe webhook handler must be idempotent on event.id — Stripe retries up to 3 days on a 5xx, and a non-idempotent handler will charge twice, ship twice, or send duplicate emails. Pinned turns that requirement into a permanent CI test from a single line in your PR description.

The claim that becomes the test

Add this line to your PR description:

Makes /webhooks/stripe idempotent on event_id.

When the PR is opened, Pinned's GitHub Action generates:

// tests/pinned/pr-42-idempotent-webhooks-stripe.test.ts
import { describe, it, expect, beforeAll } from "vitest";

const PREVIEW_URL = process.env.PREVIEW_URL;
const ROUTE = "/webhooks/stripe";
const ID_FIELD = "event_id";

describe("pinned: idempotent on " + ROUTE, () => {
  beforeAll(() => {
    if (!PREVIEW_URL) throw new Error("PREVIEW_URL required");
  });

  it("returns byte-identical response on duplicate event_id", async () => {
    const url = PREVIEW_URL + ROUTE;
    const payload = {
      [ID_FIELD]: "pinned-idem-" + Math.random().toString(36).slice(2, 12),
      // EDIT ME: add any other required fields here.
    };
    const fire = () => fetch(url, {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify(payload),
    });
    const first = await fire();
    const firstBody = await first.text();
    const second = await fire();
    const secondBody = await second.text();
    expect(second.status).toBe(first.status);
    expect(secondBody).toBe(firstBody);
  });
});

Why byte-identical?

RFC-7231 idempotency means a retry produces the same observable result. For Stripe-style handlers, that translates to: same response status + same body. Some handlers return 200 OK twice (preferred), others return 409 Conflict on the duplicate, but the responses must be deterministic across retries.

The generated test makes the assertion the strongest universal form: status equality + body equality. If your handler legitimately returns a server-generated timestamp that drifts between calls, narrow the assertion or retire the claim with pinned retire.

What this catches in production

Common ways Stripe webhook idempotency silently breaks:

  • A new code path doesn't check the dedup table before inserting
  • The dedup table TTL expires and old events get reprocessed
  • Database transactions roll back the dedup write but not the side effect
  • A refactor swaps insert-or-update logic from upsert to insert

Without a pinned test, you'd find out via a customer complaint (double charge, duplicate notification). With the pin, CI fails on the breaking commit and points to the original PR.
