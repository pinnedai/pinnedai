CURSOR

Tests for Cursor-generated code

Cursor ships PRs at 10× human velocity. Reviewers can't verify every behavioral claim against the diff — they trust Cursor's summary and merge. Pinned turns the trust step into a permanent test.

The Cursor-coding pain pattern

You give Cursor a task. It writes a PR with a description like:

Adds rate limiting to /api/users (60 req/min) to stop scraping abuse.
Adds auth check to /api/admin/billing.
Makes /webhooks/stripe idempotent on event_id.

The diff is 200+ lines across 8 files. Manually verifying each claim against the diff would take 30 minutes — too long, you merge. Six months later, an unrelated refactor by a teammate breaks one of the claims. Nobody notices until production.

What Pinned does on the Cursor PR

The Pinned GitHub Action runs the moment the PR is opened. For Cursor's three claims above, it:

  1. Parses each claim from the PR description (regex + LLM)
  2. Generates 3 Vitest files under tests/pinned/
  3. Auto-commits them to the Cursor branch
  4. Posts a PR comment listing the new pins + any unpinned risk surfaces

Now the Cursor PR isn't just a claim — it's a claim plus a test that enforces the claim forever.

What happens when a Cursor PR breaks an old claim

This is the killer scenario. Cursor refactors something six months after PR #1247, and inadvertently breaks the rate limiter on /api/users. The pinned test fails in CI with:

FAIL  tests/pinned/pr-1247-rate-limit-api-users.test.ts
  AssertionError: expected at least one 429 across 61 parallel requests, got 200,200,200,...

═══ PINNED FAILURE — paste this into Claude Code / Cursor ═══

Fix the failing pinned claim in this test file:
  Claim: "Rate-limits /api/users to 60 req/min"
  Original PR: pr-1247
  Route: /api/users
  Expected: 61 parallel requests should yield at least one 429
  Actual: got statuses 200,200,200,...

Find where /api requests are rate-limited (middleware, route handler,
or upstream proxy) and restore enforcement for /api/users at 60/minute.
Preserve all other behavior. Do not modify this pinned test file.

After fixing, re-run: npx vitest run tests/pinned/pr-1247-rate-limit-api-users.test.ts
═══════════════════════════════════════════════════════════════

The failure message is a Cursor-ready prompt. You paste it back into Cursor, it generates the fix, you commit, CI passes. The pinned test stays in your repo, ready to catch the next regression.

Pinned is the "second AI agent" workflow you already use

Many Cursor users already use a manual second-AI pattern: one AI writes the code, a second AI reviews it for bugs. Pinned automates the durable half — instead of throwaway reviews, it produces tests that stay in your repo and detect regressions automatically forever.
