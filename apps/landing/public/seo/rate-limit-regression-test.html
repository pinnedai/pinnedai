RATE LIMIT

Rate-limit regression test from a PR claim

Rate limiters are the most commonly-broken-silently security feature in production codebases. They're added in response to abuse, work for six months, then quietly break during an unrelated refactor — usually noticed only when the abuse returns. Pinned catches it.

The scenario every backend engineer has seen

A user-search endpoint gets scraped at 5000 req/min. Your team adds a rate limiter at 60 req/min and ships PR #1247 with this description:

Rate-limits /api/users to 60 req/min to stop the scraping abuse from last week.

The PR is approved and merged. The scraping stops. Everyone moves on.

Six months later, a different engineer refactors the middleware to use a new library. The library has a slightly different matcher syntax, and the rate-limit rule for /api/users is silently dropped. The scraping resumes immediately, but you don't notice until the API gets DDoSed two weeks later.

What Pinned generates from the claim

When PR #1247 is opened, Pinned commits this test:

// tests/pinned/pr-1247-rate-limit-api-users.test.ts
import { describe, it, expect, beforeAll } from "vitest";

const PREVIEW_URL = process.env.PREVIEW_URL;
const ROUTE = "/api/users";
const RATE = 60;
const WINDOW = "minute";

describe("pinned: rate-limit on " + ROUTE + " (60/minute)", () => {
  beforeAll(() => {
    if (!PREVIEW_URL) throw new Error("PREVIEW_URL required");
  });

  it("returns 429 after exceeding 60 requests/minute", async () => {
    const url = PREVIEW_URL + ROUTE;
    // Burst-parallel: fire RATE + 1 requests at once so we exceed
    // the limiter regardless of how its bucket refills.
    const statuses = await Promise.all(
      Array.from({ length: RATE + 1 }, () =>
        fetch(url, { method: "GET" }).then((r) => r.status)
      )
    );
    expect(statuses.includes(429)).toBe(true);
  });
});

Why burst-parallel?

The test fires RATE + 1 = 61 requests in parallel, not sequentially. This guarantees we exceed the rate limit regardless of how the bucket refills (token bucket, sliding window, leaky bucket — all are exceeded by a parallel burst of N+1 requests when the cap is N).

If we fired sequentially, a 60/min limit might refill enough between requests that 61 sequential requests don't hit 429. Parallel removes that ambiguity.

What variations are supported?

Pinned's parser handles several phrasings of the same claim:

  • Rate-limits /api/users to 60 req/min.
  • Rate-limit /api/users to 60 requests per minute
  • Rate-limited /api/users to 60 calls/sec
  • Rate-limits /api/users to 60 rpm (rpm / rps / rph also accepted)

Plus the LLM fallback (in CI, keyless via OIDC) handles natural-language phrasings like "the user search endpoint should throttle at 60 per minute."

Six months later, when the regression hits

When the middleware refactor drops the rate-limit rule, CI fails on the next push:
